# Aircraft End-to-End Asset Workflow

This document defines the standard workflow for taking a manually generated 2D aircraft concept image and turning it into a usable aircraft asset inside the game.

The game architecture intentionally separates:

1. **Visual asset**

   * Meshy-generated and Blender-cleaned 3D model.
   * Used only for rendering.
   * Does not determine aircraft physics.

2. **Aircraft physics / aeronautics profile**

   * Data-driven profile used by the flight dynamics engine.
   * Associated with the aircraft by `aircraftProfileId`, `aircraftName`, or asset metadata.
   * May be based on real-world data, sourced estimates, or fallback generic class values.

This workflow assumes the aerodynamics and flight dynamics framework already exists.

---

# 1. Core Principle

The aircraft mesh is only a visual skin.

A Meshy-generated model named `spitfire_mk_i.glb` should not be analyzed geometrically to derive flight behavior. Instead, it should be mapped to an aircraft profile such as:

```ts
aircraftProfileId: "spitfire_mk_i"
```

or, if exact real-world data is not available:

```ts
aircraftProfileId: "spitfire_like"
```

The flight model should use the aircraft profile data, not mesh geometry.


# 3. Naming Convention

Every aircraft should have a stable `aircraftId`.

Use lowercase snake case:

```txt
generic_trainer
spitfire_mk_i
spitfire_like
p51_mustang_like
bf109_like
heavy_bomber_baseline
classic_biplane
performance_glider
early_jet_fighter
delta_interceptor
cargo_transport
```

The visual asset and physics profile should share a common identifier when possible.

Example:

```ts
{
  assetId: "spitfire_mk_i_visual_001",
  aircraftId: "spitfire_mk_i",
  aircraftProfileId: "spitfire_mk_i",
  displayName: "Spitfire Mk I",
  modelPath: "/models/optimized/aircraft/spitfire_mk_i/spitfire_mk_i.glb",
  referenceImagePath: "/images/references/aircraft/spitfire_mk_i/source.png"
}
```

If an aircraft is inspired by a real aircraft but not intended to be exact, use a `*_like` profile.

Example:

```ts
{
  assetId: "meshy_spitfire_like_001",
  aircraftId: "spitfire_like",
  aircraftProfileId: "spitfire_like",
  displayName: "Spitfire-Like Fighter",
  modelPath: "/models/optimized/aircraft/spitfire_like/spitfire_like.glb",
  referenceImagePath: "/images/references/aircraft/spitfire_like/source.png"
}
```

# 6. Meshy 3D Generation

Codex should use the existing Meshy Pro pipeline scripts.

Do not expose `MESHY_API_KEY` to the frontend.

The API key must remain in local `.env`:

```env
MESHY_API_KEY=
```

Codex should:

1. Confirm `MESHY_API_KEY` exists.
2. Confirm the source image exists.
3. Submit the image-to-3D task.
4. Save the Meshy task response.
5. Poll until complete.
6. Download the resulting model.
7. Save the raw GLB.

Expected output:

```txt
public/models/raw/aircraft/<aircraftId>/<aircraftId>_raw.glb
public/models/raw/aircraft/<aircraftId>/meshy-task.json
public/models/raw/aircraft/<aircraftId>/meshy-metadata.json
```

If the repo has a dry-run mode, Codex should run dry-run first.

Codex should not batch-generate multiple aircraft unless explicitly instructed.

Operational notes from the first `spitfire_like` run:

* Treat an unambiguous user request to "generate one raw GLB" as approval for exactly one live Meshy create call for that aircraft only, after confirming the image exists and running the dry-run path.
* Meshy task creation is the credit-spending step. Polling and downloading the same task should not create another task, but still require network access.
* Existing Meshy scripts may default to generic `public/models/raw` output folders. For aircraft, set `ASSET_RAW_DIR`, `ASSET_CLEANED_DIR`, and `ASSET_OPTIMIZED_DIR` for the command, or otherwise ensure outputs land under `public/models/{raw,cleaned,optimized}/aircraft/<aircraftId>/`.
* Save both the script metadata (`<assetName>.meshy.json`) and the canonical workflow files (`meshy-task.json`, `meshy-metadata.json`). Do not repeat temporary signed Meshy asset URLs in public summaries.
* A multi-view reference sheet can still be used with the single image-to-3D pipeline if the user asked for image-to-3D and only provided one image. Do not split or batch-generate unless explicitly requested.

# 7. Blender Cleanup

Codex should use the existing Blender cleanup pipeline.

Input:

```txt
public/models/raw/aircraft/<aircraftId>/<aircraftId>_raw.glb
```

Output:

```txt
public/models/cleaned/aircraft/<aircraftId>/<aircraftId>_cleaned.glb
```

Cleanup goals:

* Import raw GLB.
* Confirm model orientation.
* Apply transforms.
* Set origin correctly.
* Ensure aircraft nose points along the project’s expected aircraft-forward direction.
* Remove unnecessary cameras/lights.
* Remove obvious junk geometry where safe.
* Recalculate normals.
* Simplify only if configured.
* Rename objects consistently.
* Preserve visual hierarchy if useful.
* Export cleaned GLB.

Important:

Do not modify the flight model based on the mesh geometry.

Blender cleanup is visual/runtime preparation only.

Recommended orientation check:

```txt
Aircraft local +X = right wing
Aircraft local +Y = up
Aircraft local -Z = nose / forward direction
```

If the generated aircraft points along another axis, Codex should fix the visual asset orientation during Blender cleanup.

Codex should create a cleanup report:

```txt
public/models/cleaned/aircraft/<aircraftId>/blender-cleanup-report.json
```

The report should include:

```json
{
  "aircraftId": "",
  "inputPath": "",
  "outputPath": "",
  "orientationFixed": true,
  "scaleApplied": true,
  "objectsRemoved": [],
  "objectsRenamed": [],
  "warnings": []
}
```

Orientation notes:

* The game/runtime aircraft frame is `+X` right wing, `+Y` up, and `-Z` nose/forward.
* Blender's glTF importer maps runtime `-Z` into Blender `+Y`. A correctly oriented aircraft will usually show its nose pointing toward Blender `+Y` in a top view after import.
* Meshy aircraft may arrive with nose along Blender `+X` or `-X` and up along Blender `+Z`. Render or inspect axis views before choosing the rotation.
* Record the raw orientation, baked cleanup rotation, and verification method in `blender-cleanup-report.json`.
* This orientation check is visual/runtime preparation only. It must not be used to derive aerodynamic properties.

# 8. GLB Optimization

Codex should use the existing glTF Transform pipeline.

Input:

```txt
public/models/cleaned/aircraft/<aircraftId>/<aircraftId>_cleaned.glb
```

Output:

```txt
public/models/optimized/aircraft/<aircraftId>/<aircraftId>.glb
```

Recommended optimization steps:

* Inspect GLB.
* Prune unused data.
* Weld where safe.
* Resize textures if oversized.
* Compress textures only if configured and supported.
* Avoid over-compressing early test assets.
* Validate final GLB.

Codex should write:

```txt
public/models/optimized/aircraft/<aircraftId>/gltf-report.json
public/models/optimized/aircraft/<aircraftId>/asset-metadata.json
```

Windows/npm notes:

* If `npm run asset:inspect -- --input ...` or `npm run asset:optimize -- --input ...` mis-parses arguments, call the underlying tool directly with `npx gltf-transform inspect ...` or `npx gltf-transform optimize ...`.
* If a TypeScript wrapper fails with a Windows spawn error, use the same underlying glTF Transform CLI command and record that fallback in the final report.

Recommended `asset-metadata.json`:

```json
{
  "assetId": "",
  "aircraftId": "",
  "displayName": "",
  "aircraftProfileId": "",
  "sourceImagePath": "",
  "rawModelPath": "",
  "cleanedModelPath": "",
  "optimizedModelPath": "",
  "createdAt": "",
  "visualOnly": true,
  "physicsSource": "aircraftProfile",
  "notes": []
}
```

# 10. Aerodynamic Profile Creation

Codex should create or update an aircraft physics profile associated with:

```txt
aircraftProfileId
```

This document does not define the exact aero framework fields because the project already has an aerodynamics framework.

Codex should inspect the existing profile type and populate the appropriate fields.

If the existing profile type supports metadata, store source URLs, assumptions, confidence, profile mode, and game-tuning notes directly on the profile. If it does not, add a small optional metadata block or use the existing aircraft profile sources file pattern.

The profile should be based on one of these modes:

## Mode A — Real Aircraft Profile

Use this when the generated aircraft intentionally represents a known aircraft.

Example:

```txt
aircraftId: spitfire_mk_i
displayName: Supermarine Spitfire Mk I
profileMode: real_aircraft
```

Codex should search for reliable public data and cite sources in the profile metadata.

Minimum values to research where possible:

```txt
mass / loaded weight
empty weight
max takeoff weight
wing area
wingspan
length
engine power or thrust
maximum speed
stall speed
service ceiling
rate of climb
aspect ratio, if available
```

Optional values:

```txt
moment of inertia estimates
CLmax estimate
CD0 estimate
Oswald efficiency estimate
control authority estimates
roll-rate estimates
propeller efficiency assumptions
```

Preferred source hierarchy:

1. Pilot operating handbook, flight manual, or manufacturer data.
2. Museum or aviation archive.
3. Government or military technical report.
4. Reputable aviation reference site.
5. Wikipedia or general sources only as fallback.
6. Internal game-derived estimates only when real values are unavailable.

Codex must store source notes.

Recommended source metadata shape:

```ts
sources: [
  {
    label: "Source name",
    url: "https://...",
    fieldsUsed: ["wingspan", "wingAreaM2", "loadedMassKg"],
    confidence: "high"
  }
]
```

## Mode B — Inspired-By Profile

Use this when the generated aircraft is fictional but inspired by a real type.

Example:

```txt
aircraftId: spitfire_like
displayName: Spitfire-Like Fighter
profileMode: inspired_by_real_aircraft
```

Codex should:

1. Use real-world aircraft data as a reference range.
2. Avoid claiming exact historical accuracy.
3. Create a game-stable profile.
4. Label the result clearly as inspired-by.
5. Record assumptions.

Example note:

```ts
notes: [
  "Inspired by early WWII single-engine fighters.",
  "Values are game-tuned and not intended to exactly reproduce a specific aircraft.",
  "Mesh geometry is not used for aerodynamic calculations."
]
```

## Mode C — Generic Class Profile

Use this when the aircraft is a broad gameplay class.

Example:

```txt
aircraftId: heavy_bomber_baseline
displayName: Heavy Bomber Baseline
profileMode: generic_class
```

Codex should use internally consistent estimates based on the class:

```txt
biplane
trainer
ww2_fighter
heavy_bomber
cargo_transport
glider
early_jet
delta_wing
rocket_plane
flying_wing
```

Codex should document assumptions and assign confidence as medium or low.

# 11. Derived Aerodynamic Estimates

When exact aerodynamic coefficients are not available, Codex may estimate them from known aircraft properties and game-stable defaults.

Codex should use the existing aerodynamics framework’s preferred fields.

Typical derived values may include:

```txt
aspectRatio = wingspan^2 / wingArea
meanChord = wingArea / wingspan
wingLoading = loadedWeight / wingArea
powerLoading = loadedWeight / enginePower
```

Possible game-friendly assumptions:

```txt
CLmax:
- clean light trainer: 1.3 to 1.6
- WWII fighter clean: 1.3 to 1.7
- biplane: 1.4 to 1.8
- glider: 1.2 to 1.6
- heavy bomber: 1.2 to 1.5

CD0:
- clean fighter: 0.020 to 0.035
- trainer: 0.030 to 0.050
- biplane: 0.050 to 0.080
- bomber/cargo: 0.035 to 0.060
- glider: 0.012 to 0.030

Oswald efficiency:
- conventional aircraft: 0.70 to 0.85
- biplane: 0.55 to 0.70
- heavy bomber/cargo: 0.65 to 0.80
- glider: 0.80 to 0.95
```

These are not strict rules. They are starting points for game tuning.

Codex should make conservative choices for first-pass playability.

# 12. Aircraft Profile Validation

After creating or updating an aircraft profile, Codex should run any existing profile validation.

If no validation exists, Codex should at least check:

```txt
massKg > 0
wingAreaM2 > 0
wingspanM > 0
meanChordM > 0
CLmax > CL0
CD0 > 0
stallAngleDeg is plausible
maxSpeedMps > stallSpeedMps
engine thrust or power is positive when applicable
aircraftProfileId is unique
manifest entry points to existing GLB
```

Codex should flag but not necessarily block uncertain values.

# 13. Integration with Game

Codex should wire the generated aircraft into the game without changing the physics architecture.

Expected runtime behavior:

1. Game loads an aircraft asset entry.
2. Asset entry provides:

   * `modelPath`
   * `aircraftProfileId`
   * `displayName`
3. Renderer loads the GLB from `modelPath`.
4. Flight engine loads the profile from `aircraftProfileId`.
5. Visual model follows the simulated aircraft state.
6. HUD shows aircraft display name and profile id.

The aircraft model must not calculate physics from its mesh geometry.

For the current React flight test scene, the stable pattern is:

```txt
src/assets/assetManifest.ts maps aircraftProfileId -> optimized runtimePath
AircraftController resolves visual asset by profile id
FlightTestLevel selects aircraftProfileId
AircraftProfile remains the physics source
```

Keep the geometry-only test aircraft as a fallback for profiles that do not yet have optimized GLBs.

# 17. Definition of Done

A processed aircraft is complete when:

```txt
[ ] Source 2D image exists in public/images/references/aircraft/<aircraftId>/
[ ] Meshy raw GLB exists
[ ] Meshy metadata exists
[ ] Blender-cleaned GLB exists
[ ] Blender cleanup report exists
[ ] Optimized runtime GLB exists
[ ] glTF report exists
[ ] Runtime asset manifest entry exists
[ ] Aircraft profile exists
[ ] Profile has source notes or assumptions
[ ] Profile validates successfully
[ ] Aircraft can be selected or loaded in the flight test scene
[ ] Visual model follows physics state
[ ] Flight physics come from aircraftProfileId
[ ] Mesh geometry is not used for aero calculations
[ ] Build passes
[ ] Tests pass, if applicable
```

Additional verification learned from the first processed aircraft:

```txt
[ ] Dry-run Meshy command was run before the live create call
[ ] Exactly one live Meshy create task was made for the requested aircraft
[ ] Orientation was checked visually or with an axis/sign test after cleanup
[ ] glTF optimization report includes size, bounds, triangle/vertex summary, texture size, and required extensions
[ ] Browser smoke check confirms the optimized GLB loads in the flight scene
[ ] Final report states whether generated GLBs are ignored by git and need Git LFS or .gitignore changes before sharing
```

# 18. Important Warnings

Do not allow the workflow to silently invent exact aircraft data.

If exact data is unavailable, Codex should say so and create a game-tuned estimate.

Every aircraft profile should clearly indicate one of:

```txt
real_aircraft
inspired_by_real_aircraft
generic_class
```

Every profile should include:

```txt
source confidence
assumptions
game tuning notes
```

Do not overfit physics to a single source.

Do not make the first pass unflyable in pursuit of realism.

Do not let the 3D asset pipeline modify the aerodynamics engine architecture.

Do not use the Meshy model geometry as aerodynamic truth.
